# pm5min / pm15min 全量双轨拆分设计

## 背景

当前仓库已经存在 `src/pm5min/`，但它仍然主要作为薄壳入口使用，核心实现仍然集中在 `src/pm15min/`。  
5 分钟和 15 分钟共享了大量 `data / research / live / cli` 路径，5m 逻辑主要通过 cycle/profile/offset 分支嵌入 15m 主体代码。

这种结构的问题是：

- 5m 和 15m 的职责边界不清晰
- 5m 的变更容易误伤现有 15m 稳定流程
- 很多模块同时承担“15m 主实现 + 5m 例外分支”
- 顶层入口虽然看起来有 `pm5min`，但运行心智仍然依赖 `pm15min`

本次重构的目标不是继续在同一套代码上叠加分支，而是把 15m 和 5m 彻底拆成两条独立应用线。

## 用户约束

- 采用全量双轨拆分，不接受继续在同一主体里混放
- 允许 15m 代码一起重整，不要求绝对冻结
- 对外入口也彻底分离：
  - `pm15min` 只负责 15m
  - `pm5min` 只负责 5m
- 新工作必须从当前 `main` 新开分支开始
- 放弃之前那条 5m 试验提交链，不把它作为新分支历史基础
- 重构过程中优先保证 15m 现有流程可持续验证

## 设计目标

1. 建立物理分离的 15m 和 5m 代码边界
2. 将共享层压缩到只剩“无周期立场”的基础能力
3. 让 5m 不再通过 `pm15min` 入口转发或借壳运行
4. 让 15m 移除所有 5m 特判后仍保持单轨清晰
5. 每一阶段都可单独验证、单独回退

## 非目标

- 本次设计不追求保留旧的混合入口兼容层
- 不尝试在第一阶段统一 5m/15m 所有业务实现细节
- 不把大量“看似相似但周期语义不同”的业务代码强行抽进共享层

## 方案比较

### 方案 A：整包复制分叉

将 `pm15min` 大量复制到 `pm5min`，快速做成两套目录。

优点：

- 最快得到物理分离
- 最直观

缺点：

- 重复代码会快速膨胀
- 后续 bugfix、基础能力改动需要双份维护
- 复制后仍然很难识别哪些才是真正可共享能力

### 方案 B：双应用 + 薄共享内核

建立 `pm15min`、`pm5min` 两套完整应用，将真正无周期语义的基础能力抽到新共享层。

优点：

- 边界清晰
- 15m 和 5m 各自独立演进
- 共享层语义清楚，长期维护成本可控

缺点：

- 首次拆分成本高于纯复制
- 需要明确划分哪些东西绝不能共享

### 方案 C：单包内双轨分区

继续保留 `pm15min` 主包，只在内部做 `15m / 5m` 子分区。

优点：

- 搬动少

缺点：

- 对外入口与内部职责仍然混杂
- 用户明确不希望继续“放在一起”
- 无法从根上解决 15m 被 5m 连带影响的问题

## 选型

采用方案 B：**双应用 + 薄共享内核**。

原因：

- 满足“彻底拆开”的目标
- 不会像整包复制那样把重复债务永久固化
- 能通过清晰边界保护 15m，同时给 5m 独立演进空间

## 目标结构

### 应用层

- `src/pm15min/`
  - 完整保留 15m 自己的 `core / data / research / live / cli`
- `src/pm5min/`
  - 完整拥有 5m 自己的 `core / data / research / live / cli`

### 共享层

- `src/pmshared/`
  - 只放无周期语义的基础件

### 边界规则

- `pm15min` 不得 import `pm5min`
- `pm5min` 不得 import `pm15min`
- 两边只允许 import `pmshared`
- 任何带有 cycle/profile/offset/bundle/feature/live-signal/runner 语义的模块不得进入共享层

## 共享层范围

### 允许进入 `pmshared`

- 通用时间处理
- 序列化与文件 IO 基础件
- 无周期语义的通用路径/命名工具
- 外部数据源底层 client
- 基础重试/缓存/日志小工具
- 不带周期立场的通用配置结构

### 禁止进入 `pmshared`

- cycle contracts
- feature builders
- labels
- bundle loader / active selection
- live signal / scoring / runner
- profiles
- 研究训练、实验、回测策略入口
- 带周期语义的数据调度与产物布局
- CLI 入口

## 目录迁移原则

### 15m 目录

`pm15min` 最终只保留 15m 单轨实现，不再包含任何 5m profile、5m cycle 分支或 5m CLI 默认值。

### 5m 目录

`pm5min` 最终必须拥有完整实现，不能再通过“改参数后跳回 `pm15min`”运行。

### 共享目录

共享层只接受稳定基础能力迁移，不接受“为了少写代码”而做的业务级合并。

## 第一阶段优先拆分点

优先处理最容易造成 5m / 15m 串线的模块：

1. 顶层 CLI 与包入口
2. `live/profiles`
3. `live/service`
4. `live/signal`
5. `research/config`
6. `research/bundles`
7. `data/config`
8. `core/config`

原因：

- 这些模块直接定义了运行入口、周期身份和 bundle/profile 语义
- 它们是“5m 借 15m 壳”最明显的路径
- 先拆这些，才能让后续 `data / research / live` 拆分有稳定壳体

## 迁移顺序

### Phase 0：新分支与 15m 基线保护

- 从当前 `main` 新开一条全新分支
- 不继承旧 5m 试验提交历史
- 建立 15m 基线验证集，固定当前关键入口行为

基线验证至少覆盖：

- 15m CLI 主入口
- 15m research 关键命令
- 15m live profile / signal / runner 关键路径

### Phase 1：拆入口

- `pm15min` 顶层入口只服务 15m
- `pm5min` 顶层入口只服务 5m
- 删除 `pm5min -> pm15min.cli` 的转发依赖

目标：

- 从入口心智上先完成分家

### Phase 2：建立共享层

- 新建 `pmshared`
- 仅迁移稳定基础件
- 保持业务模块暂不搬动

目标：

- 为后续双轨应用提供最薄公共地基

### Phase 3：建立 `pm5min` 完整骨架

- 建立 `pm5min.core`
- 建立 `pm5min.data`
- 建立 `pm5min.research`
- 建立 `pm5min.live`
- 建立 `pm5min.cli`

目标：

- 让 `pm5min` 先具备完整承接能力

### Phase 4：按层迁移 5m

迁移顺序：

1. `core`
2. `data`
3. `research`
4. `live`
5. `cli`

每迁完一层都执行：

- 5m 新路径验证
- 15m 回归验证
- 删除 `pm15min` 中同层 5m 分支

### Phase 5：清理 15m 中残余 5m 逻辑

- 删除 5m profiles
- 删除 5m offsets / cycle 特判
- 删除 5m 专用 bundle / signal 分支
- 删除旧兼容入口

目标：

- 让 `pm15min` 重新成为纯 15m 单轨实现

### Phase 6：加架构护栏

- 新增 import 边界约束
- 阻止 `pm15min <-> pm5min` 直接互引
- 明确只允许通过 `pmshared` 共享基础件

## 验证策略

### 每阶段都执行双验证

- 验证 15m：确认现有主流程未崩
- 验证 5m：确认它不再借用 15m 外壳

### 验证原则

- 拆一层，验一层
- 不采用“全部搬完再统一验证”
- 每一层都必须能独立回退

### 第一阶段验证重点

- `pm15min` 命令仍然只走 15m
- `pm5min` 命令不再跳回 `pm15min`
- 15m live signal / runner 关键路径保持稳定
- 5m profile / signal / bundle 不再存放于 `pm15min`

## 风险与控制

### 风险 1：15m 回归

控制：

- 先建立 15m 基线验证
- 每阶段都先回看 15m

### 风险 2：共享层过厚

控制：

- 共享层只接受无周期语义模块
- 发现周期分支就退回应用层

### 风险 3：5m 仍残留借壳路径

控制：

- 入口先拆
- 增加 import 护栏
- 在 Phase 5 清理残余兼容跳转

### 风险 4：大改难回退

控制：

- 分 phase 提交
- 每层独立验证
- 每阶段独立可回退

## 实施要求

- 所有实现从当前 `main` 新分支开始
- 不携带旧 5m 试验提交链
- 每个 phase 独立提交
- 每个 phase 完成后必须保留验证结果

## 预期完成状态

完成后应满足：

- `pm15min` 是纯 15m 应用
- `pm5min` 是纯 5m 应用
- 两边入口、配置、bundle、signal、runner、research 均独立
- `pmshared` 只承担无周期语义的基础能力
- 两边之间不存在直接 import
- 15m 不再承受 5m 实验逻辑的连带风险
