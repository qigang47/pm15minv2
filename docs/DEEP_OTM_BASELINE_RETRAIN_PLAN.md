# Deep OTM Baseline Retrain Plan

这份文档是 `docs/DEEP_OTM_BASELINE_UP_DIAGNOSIS.md` 的配套重训方案。

分工明确如下：

- `DEEP_OTM_BASELINE_UP_DIAGNOSIS.md`
  - 只负责解释当前 `deep_otm_baseline` 为什么表现不好，哪些结论已经被数据支持，哪些还只是待验证怀疑。
- `DEEP_OTM_BASELINE_RETRAIN_PLAN.md`
  - 只负责定义下一轮模型重训该怎么做，如何比较候选目标、候选因子集、候选加权方案，以及最终怎样决定是否上线。

当前版本先完成前 5 节：

1. `目标`
2. `基线定义`
3. `训练目标候选`
4. `训练样本与标签口径`
5. `样本加权策略`

后续章节会在此基础上继续补完，不再回到诊断文档里混写。

## 1. 目标

### 1.1 文档定位

这次重训不是一次普通的“再试几个 feature set”。

它要解决的是一个更具体的问题：

- 当前 `deep_otm_baseline` 在 `Down` 侧已经证明自己有一定可交易性。
- 当前 `deep_otm_baseline` 在 `Up` 侧存在明显的不稳定和失真。
- 下一代模型不能只把 `Up` 修好一点点，却明显打坏现在还能赚钱的 `Down`。

因此，这份方案的目标不是：

- 单独提高某一个分类分数
- 单独优化 `Up`
- 单独追求更多因子或更复杂模型

而是：

- 在真实可执行约束下，让 `Up` 和 `Down` 都变得更好
- 把模型重训变成一个有明确胜负标准的过程，而不是一轮“看起来更顺眼”的调参

### 1.2 总目标

本次重训的总目标是：

- 同时提升 `Up` 和 `Down` 的可交易质量
- 在不明显破坏当前有效 `Down` 的前提下，修复当前 `Up` 的主要失真
- 让最终录用标准回到真实 orderbook 结果，而不是停留在训练期分数

更具体地说，这次重训不接受下面两类“伪改善”：

- 只把 `Up` 修好，但明显伤到 `Down`
- 总体结果看起来还行，但 `Up` 继续在最关键的坏单上失真

### 1.3 主评估口径

本次重训的主评估窗口固定为：

- `最新 7 天的 orderbook 回测`

这条必须在整份方案里被视为硬约束，而不是建议。

具体含义是：

- 训练期指标只用于筛选，不用于最终拍板
- 常规概率指标只用于辅助解释，不用于最终录用
- 最终是否通过，由统一口径下的最新 `7` 天 orderbook 结果决定

也就是说，后面所有候选模型都必须接受同一条最终审判线：

- 同样的 profile 语义
- 同样的 orderbook 约束
- 同样的最新 `7` 天窗口

否则实验之间不具可比性。

### 1.4 每个实验必须汇报的结果

每个候选模型都必须按同一套格式汇报结果，不能只报一个总分。

最低要求如下。

总体结果：

- 总收益
- 总回报率
- 总交易数

`Up` 结果：

- `Up` 交易数
- `Up` 收益
- `Up` 回报率
- `Up` 胜率

`Down` 结果：

- `Down` 交易数
- `Down` 收益
- `Down` 回报率
- `Down` 胜率

分币种结果：

- `BTC`
- `ETH`
- `SOL`
- `XRP`

稳定性结果：

- 是否只靠少数几笔大单撑起来
- 是否某个币显著变好、其他币明显变差
- 是否通过大幅减少交易数来换取表面改善

如果一个实验没有同时提供以上几组结果，就不能进入下一轮比较。

### 1.5 硬通过条件

本次重训的候选模型只有在同时满足下面条件时，才可以视为“通过”。

1. 总体结果不能明显差于当前基线

- 新模型在最新 `7` 天 orderbook 下的总体收益或总体回报，不能明显差于当前 baseline。

2. `Up` 必须出现实质改善

- `Up` 收益、`Up` 回报或 `Up` 胜率中，至少要有明确改善。
- 这种改善不能主要来自“几乎不做 `Up` 了”。

3. `Down` 不能明显恶化

- 当前 `Down` 已经具备实际价值。
- 新模型不能靠牺牲 `Down` 来交换 `Up` 的局部改善。

4. 交易数不能塌陷

- 如果总交易数或 `Up` 交易数极端下降，则结果失去参考意义。
- 本次方案不接受“通过不交易来改善”的模型。

### 1.6 拒绝条件

下面几类方案，无论局部分数多好看，都应直接拒绝：

- 总体结果明显变差
- `Down` 明显恶化
- `Up` 继续持续失真，没有实质改善
- 交易数过低，导致结果不可用
- 结果主要由少数极端大单驱动
- 只在单个币上有效，而其他币明显恶化

### 1.7 本节收口

本次重训的核心判断标准是：

`候选模型是否通过，不由训练期分数决定，不由单一概率指标决定，而由最新 7 天 orderbook 下的总体结果、Up 结果、Down 结果和交易稳定性共同决定。`

## 2. 基线定义

### 2.1 为什么必须先冻结基线

如果基线不先冻结，后面所有“改进”都容易失真。

因为你现在面对的不是单一参数微调，而是多条轴同时变化：

- 训练目标可能变
- feature set 可能变
- 样本加权可能变
- 最终 live bundle 也可能变

所以本次重训必须先定义一个明确的“当前基线”，后续所有实验都只和它比较。

### 2.2 当前策略级基线

当前要冻结的策略级基线如下：

- `profile = deep_otm_baseline`
- `target = direction`
- `active_markets = btc / eth / sol / xrp`
- `offsets = 7 / 8 / 9`
- `entry_price_min = 0.01`
- `entry_price_max = 0.30`

这意味着本次重训不是为一个通用方向模型服务，而是为当前这套 deep OTM profile 服务。

如果一个候选模型的训练样本、比较口径或最终回测约束已经偏离这套 profile 语义，那么它就不再是“当前 baseline 的下一代候选”，而是在做另一个问题。

### 2.3 当前 active bundle 冻结点

截至当前工作区时间点，`deep_otm_baseline` 的 active bundle 冻结点如下：

- `btc`
  - `unified_truth0328_btc_baseline_20260328`
- `eth`
  - `unified_truth0328_eth_baseline_20260328`
- `sol`
  - `unified_truth0328_sol_baseline_20260328`
- `xrp`
  - `unified_truth0328_xrp_baseline_20260328`

从这些 active bundle 的 manifest 看，当前统一基线为：

- `feature_set = bs_q_replace_direction`
- `label_set = truth`
- `target = direction`

这点很重要，因为后面所有“重训优于当前”的判断，都必须先明确自己是在跟哪一代 bundle 比。

### 2.4 当前训练与推断栈基线

当前基线不是单模型，而是一套固定的训练和推断栈：

- 线性模型：`LogisticRegression`
- 树模型：`LightGBM`
- 融合方式：按 `Brier` 反比做 blend
- 概率保守化：使用 reliability bins 构造保守概率区间，再进入 live 决策

因此，本次第一阶段不默认“模型家族本身就错了”。

当前真正需要被重新比较的是：

- 训练目标是否错位
- 样本口径是否错位
- feature set 是否错位
- 加权方式是否错位

在没有证据之前，不把“加第三种模型”当成第一优先级。

### 2.5 当前 feature 语义基线

当前 active bundle 与最近已诊断的实盘窗口都指向同一条基线：

- `feature_set = bs_q_replace_direction`

这意味着本次重训的所有 feature 实验，默认都应视为：

- 以 `bs_q_replace_direction` 为现役基线
- 向更窄、更稳或更贴合语义的候选做比较

而不是在没有对照的前提下，再额外堆更多普通因子。

### 2.6 当前比较时必须固定不变的东西

为了保证实验可比，下面这些东西在“模型重训比较阶段”必须固定：

- 同一套 profile 语义
- 同一套最新 `7` 天 orderbook 主评估窗口
- 同一套最终 orderbook 回测约束
- 同一套币种集合
- 同一套 offset 集合

如果一个实验同时改了模型和回测约束，那它只能作为探索结果，不能作为本次重训的正式比较结果。

### 2.7 本节收口

本次重训的“当前基线”不是一句泛泛的 `deep_otm_baseline`，而是：

- 同一 profile
- 同一 active bundle 冻结点
- 同一 target
- 同一 feature 基线
- 同一最终 orderbook 评估口径

只有先把这条线冻住，后面的改进才有意义。

## 3. 训练目标候选

### 3.1 为什么训练目标必须单独比较

当前文档里的很多讨论，本质上还默认“最终方向”一定是正确训练目标。

但从当前问题结构看，这个默认并不安全。

因为 deep OTM 的真实问题不是普通的“现在往上还是往下”，而更像：

- 当前偏离发生了没有
- 这次反转有没有开始
- 它能不能在剩余时间里完成
- 翻回去之后能不能撑到结算

所以这次重训不能只在同一个旧目标上调因子，而必须正式比较训练目标本身。

### 3.2 候选目标 A：`direction`

`direction` 学的是：

- 到结算时最终是 `Up` 还是 `Down`

它的优点是：

- 和当前线上结构最接近
- 和现有 baseline 最容易直接比较
- 最适合做第一条保守基线

它的风险是：

- 容易继续学到“现在看起来更像哪边会赢”
- 容易把“局部反弹已经成立”误学成“最终一定能收成 `Up`”
- 对 deep OTM 这种终局翻回语义，不一定足够贴合

因此，在本次方案里，`direction` 只能被视为：

- `当前基线目标`

而不是：

- `默认正确目标`

### 3.3 候选目标 B：`reversal`

`reversal` 学的是：

- 从当前状态出发，这次反转到底能不能完成

这个目标更贴近 deep OTM 的真实交易语义，因为它更直接回答：

- 当前离翻盘还有多远
- 现在的反弹是不是已经足够成熟
- 这次反转是不是会走完整个窗口

它的优点是：

- 更接近你当前最痛的病灶
- 理论上更容易直接打到 `Up` 假强问题
- 更有机会把“看起来反弹了”和“真的走完了”区分开

它的风险是：

- 如果定义或切分不够严谨，容易得到一个解释很美、但实际不稳的中间目标
- 它不一定天然等于最终交易决策

因此，在本次方案里，`reversal` 应被视为：

- `独立正式候选`

而不是：

- `方向模型的小补充`

### 3.4 候选目标 C：`hybrid`

`hybrid` 不是第三种普通标签，而是一种组合思路。

它的核心思想是：

- 不强行用一个单目标解决所有问题
- 保留方向目标的总体判别能力
- 同时引入反转完成度的辅助约束

可行形式包括：

- 主模型先给出方向，再用反转模型做二次确认
- 同时训练 `direction` 和 `reversal`，再在决策层做联合筛选
- 只在主模型想做 `Up` 时，引入额外的反转完成度检查

它的优点是：

- 更符合当前真实问题结构
- 有机会兼顾现有 `Down` 优势与 `Up` 修复需求

它的风险是：

- 复杂度明显更高
- 如果没有先把单目标边界看清楚，最后很容易变成“补丁叠补丁”

因此，本次方案把 `hybrid` 放在：

- `第三层正式候选`

而不是：

- `第一轮默认方案`

### 3.5 三类目标的实验顺序

为了让实验结果能解释清楚，本次方案要求按下面顺序推进：

1. 先分别比较 `direction`
2. 再分别比较 `reversal`
3. 最后才比较 `hybrid`

不允许一开始就把三者混在一个大搜索里。

原因很简单：

- 否则你最后很难知道改善到底来自哪个目标
- 也无法分辨是目标变了有效，还是 feature / 权重 / 其他设置偶然凑巧

### 3.6 三类目标的第一轮定位

在第一轮重训里，这三类目标的职责应明确分开：

- `direction`
  - 做保守基线，回答“只改 feature 或权重时，旧目标还能不能救”
- `reversal`
  - 做语义替代候选，回答“真正贴合 deep OTM 的目标会不会更适合”
- `hybrid`
  - 做第二阶段组合候选，回答“是否需要两个目标共同工作”

### 3.7 本节收口

本次重训不默认最终方向目标一定正确，而是将：

- `direction`
- `reversal`
- `hybrid`

视为三类正式候选训练目标，并通过统一的最新 `7` 天 orderbook 评估口径决定谁更适合 `deep_otm_baseline` 的下一代模型。

## 4. 训练样本与标签口径

### 4.1 训练样本的主宇宙

本次重训的训练样本主宇宙必须是：

- `完整 canonical feature frame + label frame 对齐后的窗口集合`

而不是：

- 只使用地址上已经成交过的窗口

地址成交样本的职责应被限制为：

- 问题定位
- 病灶确认
- 失败模式举例

它不应直接承担：

- 主训练样本
- 主模型选择样本

原因很明确：

- 地址成交样本是你已经出手后的结果
- 它受当时模型、门槛、执行和仓位共同影响
- 它适合找病灶，不适合代表完整机会池

### 4.2 币种粒度

本次重训默认按币种分别训练、分别选型，不把四个币混成一个总模型榜。

原因如下：

- `BTC`
- `ETH`
- `SOL`
- `XRP`

在当前 profile 下已经表现出明显不同的结构和方向特征。

因此更合理的做法是：

- 每个币单独训练
- 每个币单独比较目标
- 每个币单独比较 feature set
- 每个币单独决定是否进入下一轮

文档里必须明确反对下面这种做法：

- 先跑一个混合四币总榜，再期待一套统一答案解决全部问题

### 4.3 offset 粒度

本次重训默认继续保留当前系统的 offset 粒度：

- `7`
- `8`
- `9`

并继续按 offset 分开训练和产出 bundle。

原因是：

- 当前 live 决策本来就是按 offset 消费模型
- 不同 offset 的可交易语义不同
- 晚段窗口和早一点的窗口，本来就不是同一种问题

如果把不同 offset 强行混到一个目标里，容易让模型学到更模糊、更难解释的折中结果。

### 4.4 标签口径

本次重训的第一轮标签口径要坚持两个原则：

1. `先保持与当前基线可比`
2. `再比较目标本身是否更适合`

因此建议如下：

- 第一轮比较中，`direction` 目标继续以当前可比标签口径为主
- `reversal` 目标使用仓库已有的 reversal 目标定义做正式对照
- `hybrid` 在第二阶段再进入正式矩阵

对训练集而言，只保留：

- `resolved = True`
- `y` 不缺失

的样本。

也就是说，不能把未结算、未对齐或标签不完整的窗口混入正式训练集。

### 4.5 label_set 的第一轮策略

当前 active baseline bundle 使用的是：

- `label_set = truth`

因此第一轮重训建议先保持：

- `label_set = truth`

作为主比较口径。

这样做的目的不是说 `truth` 一定最好，而是为了：

- 先把“训练目标”和“feature set”这两条主轴看清楚
- 不要在第一轮同时把 `label_set` 也变成主干扰项

如果后续发现：

- `direction / reversal / hybrid`
- 或 feature set / 权重

都已经有明确方向，再单独比较：

- `truth`
- `settlement_truth`

的差异会更干净。

### 4.6 与 live 语义的一致性要求

虽然训练样本主宇宙应该是完整 canonical 窗口，而不是地址成交样本，但它仍然必须服务于当前 deep OTM profile。

因此文档里需要明确：

- 训练样本的分析和汇报必须保留 deep OTM 语义切片
- 不能只看总体样本表现，而忽略当前 profile 真正会交易到的那类窗口

最低要求是：

- 分币种切片
- 分 offset 切片
- `Up / Down` 切片

在第二阶段，如果需要进一步收紧，还应考虑引入：

- 与 live 价格带更接近的上下文切片
- 更贴近 deep OTM 真实部署语义的子样本分析

但第一轮不建议同时引入过多样本裁剪，以免和目标比较、feature 比较混在一起。

### 4.7 数据质量与排除规则

本次重训的正式训练样本应明确排除下面几类内容：

- 未 resolved 的样本
- `y` 缺失的样本
- 标签对齐失败的样本
- 仅为地址分析临时补建的 fallback 行
- 任何不属于 canonical training set 构造链路的临时分析数据

这条很重要，因为地址分析里为了补齐诊断做过一些外部回补，这些数据适合解释问题，但不应混入正式训练。

### 4.8 时间边界原则

本次重训必须明确把：

- `最新 7 天 orderbook`

从训练和模型选择中隔离出去。

也就是说：

- 它是最终主评估窗口
- 不是训练窗口
- 也不是调参窗口

训练期和选择期可以使用更早的样本，但不能泄漏到最终 `7` 天。

更细的切分方法放到后续“训练与验证切分”章节详细定义，这一节先只把原则写死。

### 4.9 本节收口

本次重训的训练样本口径必须同时满足两件事：

- 不退回到地址成交样本那种窄视角
- 也不脱离当前 deep OTM profile 的实际部署语义

因此主训练宇宙应来自完整 canonical window，而正式分析和汇报必须持续保留：

- 币种切片
- offset 切片
- `Up / Down` 切片

并将最新 `7` 天 orderbook 严格保留为最终主评估窗口。

## 5. 样本加权策略

### 5.1 为什么样本加权必须进入正式方案

如果这次重训只比较目标和因子，而完全不比较样本加权，风险很大。

原因是当前真正让你痛的，不是普通错单，而是：

- 某些看起来很像机会的窗口
- 当时模型也给出较强信号
- 最后却在真实交易里伤害很大

如果训练目标默认每个样本权重都差不多，那么模型很可能继续优先照顾：

- 数量更多的普通样本
- 更容易学的普通方向关系

而不是认真处理最贵的错误。

### 5.2 当前仓库已经支持的加权能力

当前训练栈已经具备几项可直接进入实验矩阵的样本加权能力：

- 类别平衡
  - `balance_classes`
- 按波动加权
  - `weight_by_vol`
- 按波动反向加权
  - `inverse_vol`
- 对高偏离 contrarian 样本加权
  - `contrarian_weight`
  - `contrarian_quantile`
  - `contrarian_return_col`

这意味着样本加权不是一个以后才考虑的概念问题，而是当前仓库已经能直接比较的训练轴。

### 5.3 加权的基本原则

本次重训的样本加权应遵循下面几条原则。

1. 加权的目标不是“人工把 `Up` 做大”

本次方案不接受那种过于直接的思路：

- 因为 `Up` 差，所以简单给 `Up` 更高权重

这样很容易得到一个表面上更重视 `Up`、但实际把 `Down` 打坏的模型。

更合理的方向是：

- 提高真正高代价错误样本的重要性
- 提高 deep OTM 语义下更关键的样本的重要性

2. 加权不能替代最终 orderbook 评估

无论训练时怎么加权，最终判断标准都不变：

- 最新 `7` 天 orderbook

也就是说，权重只能影响训练过程，不能改变录用标准。

3. 加权实验必须和其他轴分开比较

如果同时改：

- 训练目标
- feature set
- 加权方式

最后就很难知道到底是谁起了作用。

因此样本加权应被视为一条独立实验轴，不能和其他主轴完全混在一起大搜。

### 5.4 第一轮建议比较的加权方案

为了避免实验矩阵失控，第一轮不建议做大量加权搜索，而应只比较少量、有明确含义的方案。

建议至少比较下面三类。

#### 方案 A：当前默认加权基线

这条方案的意义是：

- 保持与当前训练栈最可比
- 先看只改目标或因子时，原有加权是否已经足够

在当前仓库默认口径下，这条基线大致等价于：

- 类别平衡开启
- 波动加权开启
- 不使用额外 contrarian 放大

#### 方案 B：去掉波动加权的保守对照

这条方案的意义是：

- 检查当前按波动加权，是否在放大某些高噪音窗口
- 看模型是不是因为过度照顾高波动样本，而继续被某些假强信号说服

这条对照特别重要，因为当前 `Up` 的坏单不少本来就发生在看起来很有动量、很有波动的场景里。

#### 方案 C：适度增强 contrarian 样本权重

这条方案的意义是：

- 更认真地学习 deep OTM 最核心的那类偏离窗口
- 让“离翻盘还有距离、或刚翻回但不稳”的样本在训练里更重要

但这里建议只做：

- `温和增强`

不建议第一轮就做极端权重放大。

原因很简单：

- 权重过强容易让模型学偏
- 很可能造成 `Down` 退化

### 5.5 第一轮不建议立刻做的加权

下面这些方向可以保留为第二阶段候选，但不建议在第一轮就上。

#### 1. 直接按 `Up / Down` 做强侧向加权

风险很高，因为它太容易变成：

- 为了修 `Up`
- 人工把 `Up` 样本整体抬高

这样虽然可能短期看见 `Up` 改善，但很难保证 `Down` 不被伤到。

#### 2. 直接按地址亏损单做特殊权重

这也不适合作为第一轮方案。

原因是：

- 地址成交样本适合做病灶定位
- 但不适合直接回流成训练主权重

否则容易把“历史执行结果”直接写死进训练目标。

#### 3. 过于复杂的动态经济权重

比如：

- 同时考虑价格、剩余时间、币种、方向、历史错单强度的复合权重

这类方案不是不能做，但太适合在后期做，而不适合第一轮。

第一轮更重要的是：

- 先知道加权本身有没有帮助
- 再决定要不要做更复杂的经济权重

### 5.6 第二阶段值得考虑的加权方向

如果第一轮已经能看清目标和 feature set 的大方向，第二阶段值得考虑引入两类更贴合病灶的加权。

#### 1. 高偏离但最终未完成的样本加权

这类加权更贴近当前痛点，因为它会更认真地学习：

- deep OTM 里那些“看起来像机会，但最后没走完”的窗口

#### 2. 高代价错误样本加权

这类加权的目标不是提高某个标签的数量权重，而是提高：

- 那些一旦做错、真实伤害更大的样本

这才更符合你这次重训真正关心的问题。

但这两类加权都更适合放在：

- 目标选择和 feature set 方向已经比较清楚之后

再正式进入实验矩阵。

### 5.7 本节收口

本次重训不应把样本加权当成可有可无的小参数，而应把它视为正式训练轴。

但第一轮样本加权比较必须保持克制：

- 先比当前默认权重
- 再比去掉波动加权
- 再比适度增强 contrarian 权重

先用少量、可解释、可比较的方案回答：

`现在的模型问题，到底有多少来自目标和因子，又有多少来自训练时样本被错误地“重视”了。`

## 6. 特征方案

### 6.1 这一节的角色

这一节的任务不是重复诊断文档里的“哪些因子像病灶”，而是把这些判断翻译成一套真正可执行的重训比较方案。

所以这一节只回答四个问题：

- 这次重训先拿哪几套现有 feature set 做正式比较
- 每个币是否允许保留不同 feature set
- 新增因子应该在第几阶段进入
- 哪些事情第一轮明确不做

### 6.2 当前现役基线 feature set

当前现役基线仍然冻结为：

- `bs_q_replace_direction`

它既是：

- 当前 active bundle 的统一 feature set

也是：

- 当前诊断文档讨论的主对象

因此，后面所有 feature 实验都必须围绕它展开：

- 要么证明更窄的集合更稳
- 要么证明在补入少量关键信息后，更重组的集合更适合

而不是直接跳到一套和当前完全断裂的新体系。

### 6.3 第一轮正式比较的 feature set 候选

第一轮 feature 比较不建议自己手工临时发明太多新集合，而应优先使用仓库中已经注册、已经有清晰语义的候选。

第一轮正式比较建议固定为下面 5 套：

- `bs_q_replace_direction`
  - 当前现役宽集，作为基线
- `alpha_search_direction_live`
  - 中宽候选
- `v6_user_core`
  - 核心 live 集合
- `deep_otm_v1`
  - 窄的 OTM 语义集合
- `baseline_trim30_v1`
  - 基于当前诊断整理出的第一轮收缩候选

这些集合已经进入现有 suite spec：

- `research/experiments/suite_specs/baseline_factor_width_btc_20260401.json`
- `research/experiments/suite_specs/baseline_factor_width_eth_20260401.json`
- `research/experiments/suite_specs/baseline_factor_width_sol_20260401.json`
- `research/experiments/suite_specs/baseline_factor_width_xrp_20260401.json`

因此第一轮 feature 方案不需要先发明新的实验容器，可以直接沿用现有矩阵思路。

### 6.4 每个币独立选型

本次重训不要求四个币最后共用一套 feature set。

更合理的原则是：

- `BTC` 自己选
- `ETH` 自己选
- `SOL` 自己选
- `XRP` 自己选

允许最终结果出现：

- `BTC/ETH` 更适合某一套
- `SOL/XRP` 更适合另一套

这是因为当前诊断已经说明：

- 四个币的方向结构不同
- `Up / Down` 病灶分布不同
- 统一宽集不一定是共同最优

因此文档里必须明确反对下面这种要求：

- “四个币必须共用同一套最终 feature set”

在第一轮重训里，这种约束没有必要，反而容易把更好的币种特异结构压平。

### 6.5 feature 比较的第一原则

feature 方案比较必须遵守下面三条原则。

1. 第一轮先比现成集合，不先比大量自定义新集合

原因是：

- 这样更容易回答“问题是不是因子太宽”
- 更容易与当前 baseline 直接比较
- 更不容易把 feature 搜索做成无边界试错

2. 第一轮先比“宽度和家族结构”，不先比“单个小因子”

也就是说，优先回答：

- `64` 太不太宽
- `41 / 31 / 21 / 30` 哪个层级更对

而不是一开始就陷入：

- 某一个 RSI 变体到底删不删

3. 新增因子不和第一轮收缩实验混在一起

第一轮先回答：

- 旧因子压缩后会不会变好

只有这个问题比较清楚以后，新增因子才值得正式进入。

### 6.6 第二轮新增因子的进入方式

第二轮允许新增因子进入，但必须遵守两个约束。

1. 一次只加一组，不一起全加

建议新增因子按下面顺序进入：

- 第 1 组：剩余时间 / 触达概率
- 第 2 组：盘口斜率 / 冲击成本
- 第 3 组：微价格 / 队列失衡
- 第 4 组：不对称波动 / 波动质量

这样做的目的不是保守，而是为了让改善可解释。

2. 新增因子只能加到“已经过第一轮筛选的集合”上

也就是说，不能在一个还没证明自己合适的宽集上继续无限扩张。

正确顺序是：

- 先找到当前更合适的基底集合
- 再往这个基底上补新信息

### 6.7 第一轮明确不做的事

为了避免 feature 方案失控，第一轮明确不做下面这些事情。

- 不做 `64+` 继续扩因子
- 不做大规模暴力因子搜索
- 不做跨币统一最优 feature set 搜索
- 不做大量单因子穷举
- 不把新增因子、目标切换、样本加权同时混进同一轮实验

第一轮的目标应该很单纯：

- 先证明“当前宽集是不是问题的一部分”

### 6.8 第一轮 feature 方案的预期产出

每个币在第一轮 feature 比较结束后，都应至少产出两类结果：

1. 一个当前最优候选

- 也就是该币在当前训练目标下最值得保留的 feature set

2. 一个次优候选

- 用作下一轮继续比较和兜底

不建议第一轮结束时只保留单一冠军，因为那样容易让后续实验过早锁死。

### 6.9 本节收口

本次重训的 feature 方案，第一轮不是为了“找到终局完美因子集”，而是为了先回答一个更基础的问题：

`当前模型的问题，到底有多少来自目标错位，又有多少来自现有 feature set 太宽、太重复、太容易把局部反弹学成终局翻盘。`

## 7. 训练与验证切分

### 7.1 为什么这节必须写得很硬

如果这一节不写死，后面很容易出现一种假改善：

- 训练期分数变好
- 回测里挑出一段漂亮窗口
- 但最新一段真实交易环境一到来，结果完全变形

因此，本次重训必须把：

- 训练窗口
- 模型选择窗口
- 最终主评估窗口

三者明确分开。

### 7.2 最终主评估窗口的固定规则

本次重训的最终主评估窗口固定为：

- `运行当日向前滚动的最新 7 天 orderbook 决策窗口`

为了避免“今天、最近、最新”这类相对说法带来混淆，文档里应明确给出例子：

- 例如在 `2026-04-03` 发起正式比较时，最终主评估窗口应理解为：
  - `2026-03-27` 至 `2026-04-03`

这里的实际边界应按当次回测脚本所使用的 canonical `decision_start / decision_end` 口径落地，并统一使用 UTC 决策时间。

重要的是：

- 这一段窗口不能用于训练
- 这一段窗口也不能用于最终模型挑选时的反复调参

它只用于最后拍板。

### 7.3 推荐的三层时间切分

为了让实验结果既可比较又不泄漏，建议采用三层时间切分。

#### 第 1 层：训练窗口

训练窗口用于：

- 拟合模型
- 生成 OOF 预测
- 做第一层概率和 explainability 诊断

这层窗口应位于最终主评估窗口之前。

#### 第 2 层：模型选择窗口

模型选择窗口用于：

- 比较目标
- 比较 feature set
- 比较样本加权
- 决定谁进入最终主评估

这层窗口也必须位于最终最新 `7` 天之前，并且应紧邻最终窗口，而不是离得太远。

#### 第 3 层：最终主评估窗口

最终主评估窗口固定为：

- 最新 `7` 天 orderbook

这一层只做最终判决，不做反复调参。

### 7.4 推荐的第一轮切分方式

如果按当前仓库已有训练窗口与实验窗口的结构来落地，第一轮建议遵循下面的时间顺序：

1. 用更早的完整历史训练
2. 用最终 `7` 天之前的一段最近窗口做模型选择
3. 用最新 `7` 天做最终 orderbook 主评估

文档里不一定要先把“模型选择窗口”写死成固定天数，但至少要写死下面两条：

- 它必须紧邻最终 `7` 天之前
- 它不能和最终 `7` 天重叠

如果后续你想把这条再量化，可以在第一次冻结基线之后，把模型选择窗口固定成：

- 最新 `7` 天之前的最近 `7-14` 天

但在正式执行前，不能边跑边改。

### 7.5 训练内切分方式

在训练窗口内部，继续使用仓库已有的：

- `purged time series splits`

是合理的。

当前训练栈已经支持：

- 时间排序
- `purge_minutes`
- `embargo_minutes`

因此第一轮不建议退回普通随机切分。

为了保持与当前系统可比，第一轮建议：

- 保持 `purge_minutes = 15`
- 保持 `embargo_minutes = 0`

作为当前可比基线。

如果第一轮已经出现明确方向，再单独测试更严格的 embargo 会更干净。

### 7.6 分币种与分 offset 的切分要求

因为本次重训默认：

- 每个币单独训练
- 每个 offset 单独产出

所以时间切分也必须在各自的训练子宇宙内完成。

也就是说：

- 不能先把四个币混成一个统一时间切分，再把结果拆回去
- 不能先跨 offset 混样本，再假设每个 offset 都能共享同一时间结构

正确的做法是：

- 每个币独立
- 每个 offset 独立
- 在各自的时间序列上做 purged split

### 7.7 切分后的报告要求

每个正式实验都必须至少报告三层结果：

1. 训练内 OOF 结果

- 只作为训练质量检查

2. 模型选择窗口结果

- 用于决定谁进入最终主评估

3. 最新 `7` 天最终主评估结果

- 用于最后拍板

如果一个候选模型只有训练期和总回测结果，而没有明确区分这三层，那么它不能算正式结果。

### 7.8 本节收口

这次重训的时间切分原则只有一句话：

`训练、选择、最终评估必须彻底分开，最新 7 天 orderbook 只负责最终决定，不参与训练，也不参与反复挑参。`

## 8. 模型族与实验矩阵

### 8.1 第一轮不要把模型族也一起放开

虽然当前问题很严重，但第一轮不建议同时放开下面所有轴：

- 模型家族
- 训练目标
- feature set
- 样本加权
- 新增因子
- 二级模型

如果一上来全部放开，最后基本无法回答：

- 改善到底来自哪里

因此，第一轮实验矩阵应尽量固定：

- 模型家族先不变
- 先比较目标
- 再比较 feature set
- 再比较样本加权

### 8.2 当前模型族基线

当前训练与推断的模型族基线为：

- `LogisticRegression`
- `LightGBM`
- `inverse-brier blend`

第一轮实验矩阵建议继续沿用这一套，不把“换模型家族”当成第一主轴。

原因是：

- 当前诊断还没有证明 LR 或 LGB 的家族能力本身不够
- 当前更大的疑点仍然在目标、样本和 feature 语义

因此，第一轮矩阵的核心是：

- `同一模型族`
- `不同训练目标`
- `不同 feature set`
- `不同样本加权`

### 8.3 第一阶段实验矩阵：只比训练目标

第一阶段的目的很单纯：

- 在当前现役 feature set 和默认样本权重下，正式比较 `direction` 与 `reversal`

建议矩阵如下：

- `direction`
  - `feature_set = bs_q_replace_direction`
  - `weights = current default`
- `reversal`
  - `feature_set = bs_q_replace_direction`
  - `weights = current default`

这一阶段先不引入：

- `hybrid`
- 新增因子
- 额外 feature set 宽度比较

因为第一阶段只想回答：

- `问题是不是训练目标本身就错位了`

### 8.4 第二阶段实验矩阵：在获胜目标下比较 feature set

第二阶段只保留第一阶段表现更好的训练目标，然后比较 feature set。

建议矩阵如下：

- 训练目标固定为第一阶段优胜者
- feature set 依次比较：
  - `bs_q_replace_direction`
  - `alpha_search_direction_live`
  - `v6_user_core`
  - `deep_otm_v1`
  - `baseline_trim30_v1`

这样做的好处是：

- 不会把“目标切换”和“feature 收缩”混在一起
- 更容易看清是不是宽集本身在制造问题

### 8.5 第三阶段实验矩阵：在获胜目标 + 获胜 feature set 下比较样本加权

第三阶段只保留：

- 当前最优训练目标
- 当前最优 feature set

然后比较样本加权。

建议矩阵如下：

- `weights = current default`
- `weights = no_vol_weight`
- `weights = mild_contrarian_boost`

必要时，可以按币种分别保留不同优胜方案。

### 8.6 第四阶段实验矩阵：只在需要时引入 `hybrid`

只有在下面条件满足时，`hybrid` 才进入正式矩阵：

- `direction` 和 `reversal` 都各自展现出部分价值
- 单目标方案仍然无法同时兼顾 `Up` 和 `Down`
- feature set 和样本加权的大方向已经比较清楚

此时再比较：

- `direction baseline`
- `reversal baseline`
- `hybrid`

才有意义。

否则过早进入 `hybrid`，只会让问题解释不清。

### 8.7 每个币都走自己的矩阵

实验矩阵必须按币分开推进。

也就是说：

- `BTC` 有自己的第一阶段、第二阶段、第三阶段
- `ETH` 有自己的第一阶段、第二阶段、第三阶段
- `SOL` 有自己的第一阶段、第二阶段、第三阶段
- `XRP` 有自己的第一阶段、第二阶段、第三阶段

不要求四个币在同一阶段同时锁死同一答案。

### 8.8 第一轮明确不纳入矩阵的东西

为了控制复杂度，第一轮矩阵明确不纳入下面这些内容：

- 第三个通用模型家族
- 大规模新增因子
- side-specific 手工标签重写
- 复杂经济权重
- 执行规则同时重构

这些都可以在后续阶段进入，但不应和第一轮主矩阵一起放开。

### 8.9 本节收口

本次重训的实验矩阵核心原则是：

`一次只比较一条主轴，让每一轮实验都能明确回答一个问题。`

## 9. 评估标准

### 9.1 评估必须分层

本次重训的评估不能只看一种结果。

正确做法是分成三层：

1. 训练内评估
2. 模型选择窗口评估
3. 最新 `7` 天 orderbook 最终评估

这三层都要看，但角色不同。

### 9.2 第一层：训练内评估

训练内评估只负责回答：

- 模型有没有基本学到东西
- 概率输出有没有明显坏掉
- 某个候选是不是明显不可用

这一层建议继续保留当前训练栈已有的指标：

- `brier`
- `logloss`
- `auc`
- reliability bins

但要明确：

- 这些指标不能直接决定最终录用

它们只是第一层筛选。

### 9.3 第二层：模型选择窗口评估

模型选择窗口用于：

- 比较不同目标
- 比较不同 feature set
- 比较不同样本加权

这一层的重点不再只是分类分数，而应开始以交易结果为主。

最低要求如下。

总体：

- `pnl_sum`
- `roi_pct`
- `trades`

按方向：

- `Up pnl`
- `Up roi`
- `Up trades`
- `Down pnl`
- `Down roi`
- `Down trades`

按币种：

- 分币种总体结果
- 分币种 `Up / Down` 结果

按时间：

- 每日或分段漂移结果
- 是否存在某几天极端支撑全部结果

### 9.4 第三层：最新 7 天 orderbook 最终评估

这一层是整个方案的主评估。

文档里必须明确：

- 只有这一层可以决定是否通过
- 前两层只能决定谁值得进入这一层

在这一层，必须统一输出下列结果。

总体：

- 总收益
- 总回报率
- 总交易数

按方向：

- `Up` 收益、回报率、胜率、交易数
- `Down` 收益、回报率、胜率、交易数

按币种：

- 每个币的总体结果
- 每个币的 `Up / Down` 结果

按时间：

- 每日结果
- 累积收益曲线

按概率可信度：

- 总体 reliability
- `Up` 切片 reliability
- `Down` 切片 reliability

### 9.5 为什么必须单独看 `Up` 和 `Down`

如果只看总体，很容易被下面两种情况骗过：

- `Down` 很强，把 `Up` 的问题盖住
- `Up` 变好一点，但 `Down` 被打坏，最后总体看起来差不多

因此，文档里必须要求：

- `Up` 和 `Down` 分开汇报
- 不允许只报一个总分

### 9.6 为什么必须单独看分币种结果

因为当前 profile 下，不同币的结构明显不同。

如果只看四币混合总结果，很容易出现：

- 一个币特别好
- 另外几个币明显变差
- 总体数字却看起来还能接受

这类结果不应被视为真正通过。

### 9.7 为什么必须看稳定性

最新 `7` 天窗口本身也不算特别长，因此稳定性检查必须进入正式评估。

至少要回答：

- 是否只靠少数几笔大单撑起来
- 是否主要靠一个币支撑
- 是否通过大幅减少交易数换取表面改善

如果这些问题回答不清楚，那么结果不具上线意义。

### 9.8 评估结果的推荐呈现格式

为了便于比较，建议每个候选模型都输出同一套固定表。

最低推荐表如下：

- 表 1：总体结果表
- 表 2：`Up / Down` 结果表
- 表 3：分币种结果表
- 表 4：分币种 `Up / Down` 结果表
- 表 5：按日漂移表
- 表 6：概率可靠性表

如果后续要做自动化比较，这套表结构也最适合稳定复用。

### 9.9 本节收口

本次重训的评估标准不是：

- “谁的训练分数更高”

而是：

- “谁在最新 `7` 天 orderbook 下，总体更好、`Up` 更好、`Down` 没坏、而且结果稳定”

## 10. 通过条件

### 10.1 为什么通过条件要单独成节

如果不把通过条件单独写成规则，后面每轮实验都容易变成：

- 看起来有点改善
- 于是就继续往下做

这会让整个重训过程越来越长，但越来越不清楚到底有没有真正前进。

因此，本节要把“通过、保留、淘汰、拒绝”写成明确规则。

### 10.2 第一类条件：总体不得明显退化

无论哪个候选方案，都必须先满足：

- 最新 `7` 天 orderbook 下总体结果不能明显差于当前基线

如果一个方案：

- `Up` 好一点
- 但总体结果明显更差

那么它不能进入候选上线列表。

### 10.3 第二类条件：`Up` 必须有真实改善

这是本次重训的主问题，因此必须单独设条件。

一个候选方案只有在下面条件成立时，才能算真正改善了 `Up`：

- `Up` 收益或 `Up` 回报明显改善
- 或 `Up` 胜率改善，同时不是靠极端减少 `Up` 交易数换来的

文档里应明确反对下面这种“假改善”：

- `Up` 几乎不做了，所以亏损看起来减少

这不叫修复，只叫回避。

### 10.4 第三类条件：`Down` 不得明显恶化

当前 `Down` 已经是 baseline 的有效部分。

因此，任何候选方案如果出现下面情况，都不能通过：

- `Down` 收益明显变差
- `Down` 回报明显变差
- `Down` 交易质量显著下降

哪怕它的 `Up` 好了一些，也不能直接录用。

### 10.5 第四类条件：交易数必须仍然具有参考意义

交易数不能塌陷。

如果一个候选模型通过下面方式“改善”：

- 极端减少总交易数
- 极端减少 `Up` 交易数
- 只保留极少数高把握窗口

那么它的结果不具备足够部署意义。

因此文档里必须要求：

- 任何正式通过方案都要同时报告交易数变化
- 当改善主要来自大幅缩量时，默认进入人工复核，而不是自动通过

### 10.6 第五类条件：结果必须稳定

本次方案不接受：

- 主要由极少数单笔大赢支撑的方案
- 只在单个币上显著有效的方案
- 某几天异常好、其他时间一般甚至变差的方案

也就是说，即使一个候选模型总体结果更高，只要结构上明显失衡，它也不能直接通过。

### 10.7 推荐的结果判定层级

为了让决策更干净，建议把候选模型分成下面四档。

#### A 档：通过

同时满足：

- 总体不差于基线
- `Up` 明显改善
- `Down` 不明显恶化
- 交易数仍然具有参考意义
- 结果结构稳定

#### B 档：保留观察

满足大部分条件，但仍有一项需要复核，例如：

- `Up` 明显改善，但交易数下降偏多
- 总体更好，但某个币明显转差

#### C 档：不通过但保留研究价值

例如：

- `reversal` 目标在 `Up` 上明显有意思
- 但总体和 `Down` 暂时还不够好

这类结果可保留给下一轮组合实验，但不作为候选上线方案。

#### D 档：直接淘汰

包括但不限于：

- 总体明显变差
- `Down` 明显恶化
- `Up` 无实质改善
- 交易数失真

### 10.8 本节收口

本次重训不是在找“最有希望的故事”，而是在找“满足明确通过条件的模型”。

因此，每个候选都必须被分到：

- 通过
- 保留观察
- 保留研究价值
- 直接淘汰

四档之一，不能长期停留在“再看看”。

## 11. 实验顺序

### 11.1 为什么实验顺序必须先写死

如果不先锁定实验顺序，最容易发生两件事：

- 一次把所有东西都改了，最后不知道谁起作用
- 某个方向一时看起来有效，就把整个实验路线带偏

所以这次重训必须采用阶段式推进，而不是自由搜索。

### 11.2 第 0 阶段：冻结当前基线

第一步不是训练新模型，而是把当前基线先重新跑一遍，固定下来。

这一阶段必须完成：

- 冻结当前 active bundle
- 冻结当前 feature set
- 冻结当前 backtest spec
- 冻结当前最新 `7` 天主评估窗口

输出物至少包括：

- 总体结果
- `Up / Down` 结果
- 分币种结果
- 交易数
- 每日结果

没有这一步，后面所有“改善”都没有可信参照物。

### 11.3 第 1 阶段：先比训练目标

这一阶段只回答：

- `direction` 和 `reversal` 谁更适合当前问题

固定条件如下：

- 模型家族不变
- feature set 固定为 `bs_q_replace_direction`
- 样本加权固定为当前默认

这一阶段的输出应该是：

- 每个币在目标层面的优胜者
- 每个币的次优候选

### 11.4 第 2 阶段：在优胜目标下比较 feature set

这一阶段只回答：

- 当前问题是不是因子太宽
- 哪套 feature set 更适合每个币

固定条件如下：

- 训练目标固定为第 1 阶段优胜者
- 模型家族不变
- 样本加权仍固定为当前默认

比较对象为：

- `bs_q_replace_direction`
- `alpha_search_direction_live`
- `v6_user_core`
- `deep_otm_v1`
- `baseline_trim30_v1`

### 11.5 第 3 阶段：在优胜目标 + 优胜 feature set 下比较样本加权

这一阶段只回答：

- 当前训练时是不是“重视错了样本”

固定条件如下：

- 训练目标固定
- feature set 固定
- 模型家族固定

比较对象为：

- 当前默认权重
- 去掉波动加权
- 适度增强 contrarian 权重

### 11.6 第 4 阶段：只在需要时引入 `hybrid`

只有在下面情况同时满足时，才进入这一阶段：

- 单独 `direction` 不够好
- 单独 `reversal` 也不够全面
- 两者各自展示出不同价值

这一阶段的目标不是重新洗牌，而是回答：

- 是否需要双目标共同工作，才能同时兼顾 `Up` 和 `Down`

### 11.7 第 5 阶段：再补新增因子

只有在前 4 阶段已经基本清楚方向后，新增因子才进入正式矩阵。

顺序建议固定为：

1. 剩余时间 / 触达概率
2. 盘口斜率 / 冲击成本
3. 微价格 / 队列失衡
4. 波动质量类信息

每次只加一组，不能一次全加。

### 11.8 第 6 阶段：最终复核与定版

这一阶段的任务不是再找更好，而是确认：

- 候选模型是否在最新 `7` 天 orderbook 下稳定成立
- 是否需要按币种分别落地不同最终方案
- 是否存在某些币暂时不应该升级

### 11.9 本节收口

本次重训的实验顺序必须坚持：

- 先目标
- 再 feature
- 再加权
- 再 hybrid
- 最后才补新增因子

只有这样，每一轮实验的结论才有解释力。

## 12. 输出与决策

### 12.1 每个阶段都必须产出明确结论

这次重训不接受“跑完一堆实验，再凭感觉总结”。

每个阶段结束后，都必须形成明确输出：

- 哪些候选通过
- 哪些候选保留
- 哪些候选淘汰
- 下一阶段只允许哪些候选进入

### 12.2 最低输出物要求

每个正式实验至少应产出下面这些对象。

训练侧：

- `training_run`
- offset 级 summary
- 训练期 metrics
- reliability bins

回测侧：

- backtest summary
- trades / rejects / decisions
- offset summary
- 每日结果

比较侧：

- 候选与基线的对比表
- `Up / Down` 对比表
- 分币种对比表
- 是否进入下一轮的结论

### 12.3 每个阶段的决策动作

建议把每个阶段的决策动作固定成下面的模板。

#### 阶段结束时必须回答

1. 这一阶段回答了哪个问题
2. 哪个候选目前最好
3. 哪个候选是备选
4. 哪些候选应当淘汰
5. 下一阶段只保留哪些组合

这样做的目的，是让整个重训流程形成真正的“漏斗”，而不是越跑越宽。

### 12.4 最终候选的落地形式

最终结果不一定是：

- 四个币共用同一个模型目标
- 四个币共用同一个 feature set

更现实的结果可能是：

- 某些币保留 `direction`
- 某些币转向 `reversal`
- 某些币最终采用 `hybrid`
- 某些币使用不同的 feature set

这在本次方案里是允许的，只要：

- 主评估窗口下结果更好
- `Up` 与 `Down` 都满足通过条件

### 12.5 上线前最后一道决策门

任何候选模型在进入上线准备前，都必须再经过一次统一确认：

- 当前最新 `7` 天 orderbook 是否仍然成立
- 与冻结基线相比是否仍然满足通过条件
- 是否存在某个币需要单独延后升级

如果这一步不过，就不进入上线准备。

### 12.6 与诊断文档的关系

重训方案文档不再重复诊断细节。

它只保留两类来自诊断文档的输入：

- 已被支持的病灶结论
- 当前最值得验证的假设

所有“为什么会这样”的长篇解释，留在诊断文档。
所有“接下来怎么做”的规则与顺序，留在本方案。

### 12.7 本节收口

这份重训方案最终要产出的，不是一堆实验记录，而是一条明确的决策链：

- 哪条路被证明无效
- 哪条路被证明更好
- 哪个模型能进入下一阶段
- 哪个模型能成为下一代候选

只有这样，这份方案才真正完成了从“诊断笔记”到“重训方案”的转换。
